# Day 01：这套「学习笔记 + YouTube + 博客 + 自动化」工作流要解决什么问题？

> 本文同时是：
> - 一篇介绍这套工作流的博客；
> - 第 1 支介绍视频的脚本草稿；
> - 之后每一期可以复用的模板雏形。

---

## 🧾 一、视频信息（给自己看的元数据）

- **视频标题（暂定）**：  
  「我打算用一套“学习笔记 + YouTube + 博客 + 自动化”的流程，长期输出学习成果」

- **目标观众**：  
  - 有学习习惯，但笔记分散、难以复盘的人  
  - 想做 YouTube 或 B 站输出，但总觉得「没时间剪、没脚本、不知道做什么」的人  
  - 想把自己的学习过程“顺便”沉淀成博客 / 文档的人

- **观众看完能得到什么？**  
  - 理解这条工作流的大致结构  
  - 知道它能帮自己解决哪些痛点  
  - 如果感兴趣，可以跟着后续视频一起把整套工作流搭起来

- **视频核心一句话（方便你口播 / 写简介）**：  
  > 我在搭建一条从「学习笔记」到「博客 + YouTube 视频 + 自动部署」的完整工作流，让每一次学习都顺便变成长期可复用的数字资产。

## 🎬 二、口播脚本（录视频时可以按段落说）

> 提示：  
> - 你可以一段一段地录，后期再简单剪掉空白；  
> - **中括号**里的内容是对镜头/演示的提示，不需要念出来。

### 1. 开场（Hook + 自我介绍）

> 时长建议：30～60 秒

- 「大家好，这期视频我想和你分享一条我正在搭的一条工作流。」
- 「简单说，就是把我平时的学习笔记，自动变成一篇博客文章，再顺手变成一条 YouTube 视频。」
- 「然后用 GitHub Actions 和一台云服务器，把这些内容自动部署到一个网站上。」
- 「如果你也在做学习笔记、想做长期输出，这条工作流可能对你有帮助。」

[镜头提示：可以简单展示一下这个项目的目录结构，或者简单切到浏览器，给出你现在看到的这个文档站。]

---

### 2. 痛点：为什么我要折腾这条工作流？

> 时长建议：1～2 分钟  
> 这里建议你替换成**你自己的真实情况**。

可以按要点自由发挥，比如：

- 「我平时其实也会看很多课程、文章，但笔记经常散落在 Notion、各种 md 文件、甚至微信收藏里。」
- 「真正要用的时候，又找不到当时整理过的东西，等于白学。」
- 「另外我也想做视频输出，但每次开录就会卡在：今天录什么？脚本在哪？剪完之后就懒得再整理成文章。」
- 「久而久之，学了很多，看起来很忙，但没什么东西真正沉淀下来。」

然后接一个过渡：

- 「所以我就想试一下，能不能搭一条流程：只要我老老实实写好一篇笔记，这篇笔记就可以直接变成一支视频、变成一篇博客、同步到一个自己的站点里。」

---

### 3. 目标：这条工作流长什么样？

> 时长建议：1～2 分钟

可以大概这样说：

- 「这条工作流的大致构成是这样的：」
- 「本地这边，我用的是 Windows + VSCode + Git + MkDocs，再配一个叫 Material 的主题。」
- 「远端这边，我会用一个 GitHub 仓库，配一条 GitHub Actions 的自动构建流程，把静态文件部署到一台云服务器上。」
- 「内容这块，核心就是一份 md 笔记，然后同时作为博客文章和视频脚本来用。」

[镜头提示：  
- 可以切到 `mkdocs.yml`，简单给观众看一下 nav 结构；  
- 再切到 `docs/videos/workflow/day01.md`，给观众看一下你现在正在写的这篇内容。]

---

### 4. 流程概览：从学习到上线，一共拆成几步？

> 时长建议：1～2 分钟

你可以把 README 里的 9 个步骤，用更口语化的方式说出来，例如：

1. 「首先是学习和做原始笔记，可能会比较乱，没关系。」
2. 「第二步是把这些原始笔记，整理成一篇结构化的文章，同时顺手把视频脚本的结构搭出来。」
3. 「第三步是准备封面的构想和 Prompt。」
4. 「第四步是录制和剪辑视频，这部分我也会记录我自己怎么简化这件事。」
5. 「第五步是生成标题、描述和标签。」
6. 「第六步是上传到 YouTube。」
7. 「第七步是回到笔记里，把视频链接挂上去，让笔记和视频互相指向。」
8. 「第八步是推送到 GitHub，让 Actions 自动构建好网站，再同步到云服务器。」
9. 「第九步是反过来，在视频的描述里补上博客文章的链接，这样就形成一个闭环。」

最后来一句总结：

- 「这套流程听起来步骤挺多的，但我们会一小块一小块搭起来，先跑通最小闭环，再慢慢优化。」

## 📚 三、博客版正文（给读者看的版本）

> 这一部分是给「只看文字不看视频」的人看的，可以和上面的脚本内容保持一致，但可以写得更完整一点。

### 1. 背景：学习很多，却难以沉淀

这里可以写成完整的叙述，比如：

- 你平时是怎么学习的（看视频、看书、上课……）；
- 笔记现在分散在哪些工具里；
- 过去有哪些「想做内容输出但没坚持下去」的经历；
- 你为什么觉得「需要一条能长期使用的工作流」，而不是一次性的项目。

### 2. 这条工作流的设计原则

可以列几点你自己的思考，比如：

- 「只要写好一篇笔记，就能顺手变成多种输出形式」；
- 「尽量用开源 / 跨平台的工具，方便迁移和备份」；
- 「所有东西最终都能落在一个 Git 仓库里，方便版本管理」。

### 3. 技术栈概览

- 本地：
  - Windows + VSCode
  - Git
  - MkDocs + Material 主题
- 远端：
  - GitHub 仓库
  - GitHub Actions（自动构建静态站点）
  - 一台 Linux 云服务器（只做静态文件托管）
- 内容：
  - `docs/notes` 下的学习笔记
  - `docs/videos/...` 下和视频对应的文档

### 4. 流程概览（文字版）

可以把你刚才口播的 9 步流程，用更详细一点的方式写出来。
